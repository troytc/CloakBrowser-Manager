<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host FS → container `/data/profiles` volume | Named volume owned by host UID; container needs Chrome to be able to write (cookies, SingletonLock cleanup). UID mismatch = silent Chrome write failures. |
| Shell env → Python process | docker-compose.yml env block is the seam where .env values enter the container; fail-closed startup (Plan 03) reads os.environ from here. |
| .env.example → .env | Git-tracked documentation. .env is NOT tracked (must be in .gitignore). Placeholder values must NOT be usable in production. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-11 | Denial of Service | UID mismatch on mounted volume blocks Chrome cookie flush (Pitfall 10) | mitigate | chown -R ${CHROME_UID:-0}:${CHROME_UID:-0} /data/profiles in entrypoint, BEFORE any read/write attempts. set -e aborts startup on chown failure — loud failure, not silent. |
| T-01-12 | Information Disclosure | Operator commits real secrets to .env | accept | .env.example is tracked; .env is git-ignored (pre-existing .gitignore rule assumed). Sample values are obviously placeholder (change-me-...). If .gitignore is missing .env, call it out in verification. |
| T-01-13 | Spoofing | Missing MAIN_APP_API_KEY / VIEWER_SECRET allows any caller to hit machine routes (Phase 2 surface) | mitigate | Plan 02 declares env vars with empty-string fallback; Plan 03 enforces fail-closed. The empty string triggers fail-closed startup. |
| T-01-14 | Elevation of Privilege | Entrypoint runs as root; future USER 1000 switch silently breaks chown | accept | D-14 keeps USER root; future flip requires moving chown to pre-build or setuid helper. Document in Dockerfile comments (out of scope for Plan 02, noted in DEFERRED in CONTEXT.md). |
| T-01-15 | Information Disclosure | .env.example sample values get copied to production unchanged | mitigate | Placeholder values are `change-me-to-a-long-random-string` / `http://localhost:3000` etc. — non-functional. Fail-closed startup (Plan 03) catches `change-me-...` if it reaches production because VIEWER_SECRET would validate as a string; acceptable because the fail-closed check only tests non-empty, not strength. Stronger check is v2 hardening. |

**Controls applied:** set -e on chown; empty-string fallback for required secrets; placeholder-only sample values in .env.example; .env.example tracked in git so new developers see the full surface.
</threat_model>
