<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin → FastAPI `/api/*` | Authenticated via AuthMiddleware (bearer/cookie) — any input crossing this boundary is untrusted user data before Pydantic validates |
| Python → SQLite | Parameterized-query boundary — never string-concatenate template fields into SQL |
| Template → Profile snapshot | Blueprint JSON copied into profile columns at profile-create time; template is never re-read on wake (TMPL-05) — mutation of template MUST NOT retroactively mutate profile |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | Blueprint JSON field values crossing into SQLite | mitigate | TemplateBlueprint Pydantic model with Literal unions for platform/human_preset/color_scheme enforces typed shape before any DB write. All DB access uses `?` placeholders (existing pattern). |
| T-01-02 | Tampering | SQL injection via vendor_type or template_id in helper calls | mitigate | All helpers in backend/database.py use parameterized queries with `conn.execute("... WHERE x = ?", (val,))`; no f-string interpolation of user input into SQL. |
| T-01-03 | Information Disclosure | clipboard_sync defaulting True lets credentials leak from vendor portal clipboard to Main App | mitigate | D-18: flip default to False at 5 touchpoints (schema DEFAULT 0, ProfileCreate, ProfileResponse, coerce_clipboard_sync fallback, TemplateBlueprint default). field_validator enforces null→False coercion. |
| T-01-04 | Denial of Service | Template delete cascades into live profiles | mitigate | D-06 DB-layer half: profiles.template_id REFERENCES vendor_templates(id) ON DELETE RESTRICT. PRAGMA foreign_keys=ON (already enforced). App-layer 409 lands in Plan 04. |
| T-01-05 | Denial of Service | Concurrent profile create produces duplicate (vendor_type, vendor_connection_id) pairs | mitigate | D-05: CREATE UNIQUE INDEX on profiles(vendor_type, vendor_connection_id). DB-level safety net; app-layer upsert is Phase 2. |
| T-01-06 | Denial of Service | Schema migration wipe destroys production data | accept | D-08: explicit decision that old schema is dev/test only; no production deployment with new shape exists. Wipe logs row + directory counts at WARNING level so the destruction is visible. |
| T-01-07 | Tampering | vendor_type slug contains spaces/uppercase/Unicode → breaks downstream URL routing | mitigate | Field(pattern=r"^[a-z0-9_-]+$", min_length=1, max_length=64) on VendorTemplateCreate.vendor_type — rejects non-slug values with Pydantic 422 (Pitfall 6). |
| T-01-08 | Tampering | Legacy POST /api/profiles path collides with UNIQUE(vendor_type, vendor_connection_id) on empty-string defaults | mitigate | Pitfall 2 fix: inject `vendor_type = "__legacy__"` + `vendor_connection_id = str(uuid.uuid4())` in create_profile() when caller does not provide them. Each legacy call gets a distinct pair. |
| T-01-09 | Tampering | Blueprint round-trip loses fields when Pydantic defaults change later | mitigate | Pitfall 3: always emit with `model_dump_json()` (no exclude_defaults). On read, use `model_validate(json.loads(...))`. Pattern locked in create_template / update_template helpers. |
| T-01-10 | Information Disclosure | Unbounded launch_args list lets an admin inflate blueprint JSON size | accept | Single trusted admin surface; DB write per template is bounded by admin session; adding Field(max_length=32) is low-value belt-and-suspenders. Revisit if it becomes an issue. |

**Controls applied (summary):** FK ON DELETE RESTRICT on profiles.template_id; UNIQUE INDEX on (vendor_type, vendor_connection_id); Pydantic Literal enums and regex pattern on vendor_type; clipboard_sync default False + field_validator null-coercion; parameterized queries throughout; destructive migration is idempotent and logged.
