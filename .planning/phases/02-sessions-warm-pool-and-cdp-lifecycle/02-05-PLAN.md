<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WS connection accepted -> count not yet incremented | Race window: idle timer could fire and stop the just-attached profile. Mitigated by mutating count BEFORE on_attach |
| WS disconnect -> count not yet decremented | Race window: status reads stale count. Acceptable — count is eventually consistent |
| Lifespan shutdown -> idle timer firing during cleanup | Mitigated by sm.shutdown() before browser_mgr.cleanup_all() |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Denial of Service | Idle timer kills profile during legitimate WS connect (Pitfall 3) | mitigate | Count is incremented INSIDE browser_mgr._lock BEFORE sm.on_attach is called; sm.on_attach cancels any pending idle task; even if cancel loses the race, the idle task's defense-in-depth re-check (Plan 02-04) sees the count > 0 |
| T-02-13 (defense in depth) | Denial of Service | Double-decrement on count race (e.g., proxy raises in finally cleanup) | mitigate | max(0, count - 1) clamps below zero — prevents negative counts from falsely marking the profile as "more idle than possible" |
| T-02-14 | Denial of Service | Lifespan shutdown timing — idle timers fire on closed contexts | mitigate | sm.shutdown() drains _idle_tasks BEFORE browser_mgr.cleanup_all (RESEARCH §8) |
</threat_model>
